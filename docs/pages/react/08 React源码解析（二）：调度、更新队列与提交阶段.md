## React源码解析（二）：调度、更新队列与提交阶段

> 如果说 Fiber 解决的是“结构和可中断”的问题，那么调度、更新队列和 commit 流程，解决的就是“更新到底怎么被安排、合并、计算并真正落到页面上”。

<!-- more -->

## 一、React 的一次更新是从哪里开始的？

常见入口有这些:

+ `setState`
+ Hook 的 `dispatch`
+ `forceUpdate`
+ Context value 变化
+ 根节点重新渲染

无论入口是什么，最终都会变成:

> 给某个 Fiber 节点挂上一条更新，然后把这次更新向上标记到根节点，等待调度执行。

## 二、什么是更新对象和更新队列？

React 不会在你调用 `setState` 的那一刻立刻改掉最终渲染结果，而是先创建一条更新记录。

这条更新里通常包含:

+ 更新动作
+ 优先级信息
+ 可能的 payload
+ 关联时间信息

多个更新会进入更新队列，等到合适的调度时机再统一处理。

这也是为什么 React 的状态更新经常被感知为“异步感”。

## 三、为什么函数式更新能拿到最新值？

例如:

```jsx
setCount(prev => prev + 1)
```

这里不是直接读取当前闭包里的 `count`，而是在 React 处理更新队列时，把上一个计算得到的状态依次传给每一条更新函数。

所以它比:

```jsx
setCount(count + 1)
```

更稳，因为不会过度依赖当前渲染闭包里的旧值。

## 四、React 为什么要做优先级调度？

因为不同更新的重要性不同。

例如:

+ 用户输入需要尽快响应
+ 大列表筛选结果可以稍后一些
+ 某些后台刷新可以更低优先级

如果所有更新都一视同仁，页面就很容易出现:

+ 输入卡顿
+ 滚动掉帧
+ 大计算阻塞交互

所以 React 会为更新分配优先级，并按优先级安排执行顺序。

## 五、什么是 Lane？

在较新的 React 中，优先级不只是一个简单数字概念，而是用 `Lane` 模型来表达更新优先级和批次。

你可以把 `Lane` 理解为:

+ 一组优先级通道
+ 不同更新可以落在不同 lane 中
+ React 会根据 lane 决定先处理哪些工作

面试里通常不要求你背位运算细节，但知道它是 React 18 调度体系的重要基础会比较加分。

## 六、Scheduler 在 React 里扮演什么角色？

可以把它理解成“任务安排者”。

它的目标不是直接更新 DOM，而是帮助 React 决定:

+ 哪些任务先做
+ 哪些任务可以稍后做
+ 当前时间片是否该让出主线程

所以 Fiber 解决“能拆”，Scheduler 解决“怎么排”。

## 七、render 阶段内部大致在做什么？

大致可以拆成两步理解。

### 1. beginWork

主要任务:

+ 根据新 props / state 计算子节点
+ 决定子 Fiber 如何生成或复用
+ 向下遍历

### 2. completeWork

主要任务:

+ 向上回收结果
+ 生成或整理副作用标记
+ 处理宿主节点准备工作

可以粗略理解为:

+ `beginWork` 偏向“往下算”
+ `completeWork` 偏向“往上收”

## 八、什么是副作用标记？

React 在 render 阶段不会直接操作 DOM，所以需要记录“提交阶段该做什么”。

常见副作用包括:

+ 插入
+ 更新
+ 删除
+ ref 处理
+ 布局 effect
+ 被动 effect

这些信息会在 Fiber 上形成标记，供 commit 阶段消费。

## 九、commit 阶段大致分哪几步？

可以分成三个理解层次:

### 1. before mutation

提交前的准备阶段。

### 2. mutation

真正执行 DOM 变更:

+ 插入
+ 删除
+ 更新属性

### 3. layout

DOM 变更完成后执行布局相关逻辑:

+ `useLayoutEffect`
+ 类组件部分生命周期
+ ref 相关处理

而 `useEffect` 对应的被动副作用，通常会在稍后的时机处理。

## 十、`useEffect` 和 `useLayoutEffect` 在 commit 中的区别

### `useLayoutEffect`

+ 更靠前
+ DOM 变更后立刻同步执行
+ 浏览器绘制前完成

### `useEffect`

+ 属于被动 effect
+ 通常不会阻塞绘制
+ 更适合大多数副作用

这就是为什么二者虽然都叫 effect，但时机差别很大。

## 十一、为什么 React 可以丢弃一次 render？

因为 render 阶段主要是在内存中构建 `workInProgress` 树。

如果在这个过程中来了更高优先级更新，React 可以:

+ 中断当前低优先级工作
+ 去处理更重要的更新
+ 甚至直接废弃刚才那次尚未提交的 render 结果

只要没进入 commit，界面就还没真正被改动。

## 十二、为什么 Hooks 必须按顺序调用？

因为函数组件对应 Fiber 上会维护一个 Hook 链表或等价的顺序结构，React 是按调用顺序依次读取每个 Hook 状态的。

如果顺序不稳定:

+ 当前这个 Hook 取到的状态可能就错位
+ effect、memo、state 都可能串位

所以本质不是语法洁癖，而是实现约束。

## 十三、`setState` 到底是同步还是异步？

严格来说，这题不能简单回答“同步”或“异步”。

更准确的说法是:

+ 调用 `setState` / Hook dispatch 这件事本身是同步发生的
+ 但新状态何时被渲染和提交，要服从 React 的调度和批处理机制

所以大家体感上常说它“异步”，其实是:

> 更新结果不是立刻在当前渲染上下文中可见。

## 十四、React 源码题怎么答才不空？

建议用这条链路:

1. 更新从 `setState` / Hook dispatch 进入
2. 生成更新对象，进入更新队列
3. 标记 lane，调度到根节点
4. render 阶段构建 `workInProgress`
5. 生成副作用标记
6. commit 阶段分批提交 DOM 和 effect

只要这条链路能说顺，大部分“更新原理”题都不会太虚。

## 十五、一段适合面试现场的总结话术

> React 的更新不是调用 `setState` 后立刻同步改 DOM，而是先生成更新对象进入队列，再根据优先级通过调度器安排执行。render 阶段会基于当前 Fiber 树构建 `workInProgress`，这个过程可以中断和重做，并在 Fiber 上记录副作用标记；commit 阶段再一次性把 DOM 变更、ref 和各类 effect 提交出去。React 18 中进一步通过 Lane 模型表达优先级，所以整个更新系统既能保证最终一致性，也能兼顾交互流畅度。
