## React 18新特性与并发更新

> React 18 已经是现在面试中的标准范围。很多题都不要求你把源码背下来，但至少要理解并发更新、自动批处理、过渡更新这些概念。

<!-- more -->

## 一、React 18 的核心升级是什么？

一句话可以这样说:

> React 18 的重点不只是新 API，而是把更新调度进一步做成“可中断、可区分优先级、可渐进渲染”的模型。

高频关键词:

+ 并发渲染能力
+ 自动批处理
+ `useTransition`
+ `useDeferredValue`
+ 新的根 API
+ 对流式 SSR 的增强

## 二、什么是并发渲染？

这题很容易被说错。

并发渲染不等于:

+ 多线程
+ 同时真的并行执行多个 render

更准确的理解是:

> React 可以把一次渲染工作拆分成可中断的小任务，并根据优先级决定先做什么、后做什么，必要时还能丢弃过时渲染结果。

它解决的核心问题是:

+ 用户输入等高优先级任务不能总被大渲染阻塞
+ 渲染工作应该更有弹性

## 三、为什么说并发渲染是“能力”，不是“默认所有组件都并发”？

因为 React 18 提供的是一套更强的调度基础设施，不代表你写的每次更新都会自动表现得完全不同。

只有在某些特定场景下，例如:

+ transition 更新
+ 可中断渲染
+ Suspense 协调

你才会更明显感受到并发能力带来的区别。

## 四、React 18 的根 API 为什么改了？

旧写法是:

```jsx
ReactDOM.render(<App />, root)
```

新写法是:

```jsx
import { createRoot } from "react-dom/client"

const root = createRoot(container)
root.render(<App />)
```

这是因为新 root API 才能启用 React 18 的新特性体系。

## 五、自动批处理是什么？

React 会把同一个时机内的多个状态更新合并，减少重复渲染。

早期版本里，批处理主要发生在 React 事件中；React 18 则把范围扩展到了更多异步场景。

例如:

+ `setTimeout`
+ Promise 回调
+ 原生异步流程

都可以被自动批处理。

### 它带来的意义

+ 更少的重复渲染
+ 更一致的更新行为

## 六、`flushSync` 是什么？

有时你不想等批处理和调度，希望某次更新立刻同步生效，可以使用 `flushSync`。

但这通常是一个“逃生口”，不应滥用。

适合场景:

+ 某些必须立刻读取最新 DOM 的同步操作

不适合场景:

+ 当作日常更新方式

## 七、`useTransition` 是什么？

一句话:

> `useTransition` 允许你把某些更新标记成非紧急更新，让 React 优先响应更关键的交互。

典型场景:

+ 输入框输入时，列表筛选结果很重
+ 页面切换时，先保证点击和输入反馈流畅

它通常返回:

+ `isPending`
+ `startTransition`

## 八、`startTransition` 和普通更新的区别

普通更新:

+ 默认按较高优先级处理
+ 更适合输入、点击反馈这类立即交互

transition 更新:

+ 优先级更低
+ 可以被中断
+ 如果过时，旧的中间渲染结果可以直接丢弃

所以面试里可以概括为:

> transition 更新不是不要更新，而是“可以晚一点、必要时可以重来”。

## 九、`useDeferredValue` 是什么？

它不是给“逻辑延时”用的，而是让某个值在渲染层面延后追上最新值。

高频理解:

+ 输入框的值先立刻更新
+ 依赖该值的大列表可以稍后更新

它比手动 `debounce` 更偏“渲染优先级控制”，而不是“时间维度上的定时器控制”。

## 十、React 18 中为什么 `StrictMode` 看起来更严格了？

开发环境下，`StrictMode` 会有意地重复执行一些流程，用来帮助发现不安全副作用。

常见表现:

+ 函数组件执行两次
+ effect 挂载和清理多跑一轮

目的不是整你，而是暴露这些问题:

+ 副作用不纯
+ 没有正确清理
+ 写法依赖“只执行一次”的偶然性

## 十一、React 18 对 SSR 带来了什么增强？

可以从这几个关键词回答:

+ 流式渲染
+ Suspense 配合服务端分块输出
+ 更好的 hydration 协调

以前 SSR 更像“整页准备好再一起发”，React 18 更强调“能先发的先发，边准备边输出”。

## 十二、什么是 hydration？

服务端把 HTML 先输出给浏览器后，客户端 React 还需要接管这些已有 DOM，并给它们绑定事件、恢复交互能力，这个过程就叫 hydration。

面试里可以顺带补充:

+ SSR 解决首屏和 SEO
+ hydration 解决“静态 HTML 如何变回可交互应用”

## 十三、React 18 面试中常见追问

### 1. 并发渲染会不会让数据乱掉？

不会。虽然 render 阶段可中断、可重做，但 commit 阶段仍然是受控提交的，React 会保证界面最终一致性。

### 2. 并发更新是不是完全无感升级？

不是。业务代码里如果存在不纯副作用、依赖同步执行顺序、错误的 effect 清理逻辑，升级后更容易暴露问题。

### 3. 所有项目都必须手动用 `useTransition` 吗？

不是。它适合特定交互性能场景，不是默认必须上。

## 十四、一段适合面试现场的总结话术

> React 18 最大的升级是并发渲染能力和更现代的调度模型。它并不是多线程，而是让渲染工作可以按优先级拆分、可中断、可恢复。与此同时，React 18 还带来了自动批处理、`useTransition`、`useDeferredValue`、新 root API 以及更强的流式 SSR 支持。实际项目里，这些能力的意义主要在于让高优先级交互更流畅，同时也要求副作用写法更规范。
