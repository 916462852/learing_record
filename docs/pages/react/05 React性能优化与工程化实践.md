## React性能优化与工程化实践

> React 性能优化是高频面试区，但真正好的回答不应该只背 `memo`、`useMemo`，而是要知道“组件为什么会重新渲染”以及“优化到底在优化什么”。

<!-- more -->

## 一、React 组件为什么会重新渲染？

先把这个问题讲清楚，后面的优化才有意义。

常见触发原因:

+ 组件自己的 state 变化
+ 父组件重新渲染
+ Context 值变化
+ 外部 store 订阅值变化

需要注意:

> 父组件重新渲染时，子组件默认也会重新执行。

这里的“重新执行”不一定等于真实 DOM 一定大改，但组件函数本身通常会重新跑一遍。

## 二、React 性能优化的核心目标是什么？

本质上主要优化三件事:

+ 减少不必要的渲染计算
+ 减少不必要的真实 DOM 提交
+ 减少主线程长时间阻塞

所以不要把优化理解成只有一种方式。

## 三、`React.memo` 是什么？

`React.memo` 是函数组件层面的记忆化优化。

它会对 `props` 做浅比较，如果前后没变，可以跳过本次组件渲染结果计算。

适合场景:

+ 纯展示组件
+ 渲染成本高
+ `props` 相对稳定

不适合场景:

+ `props` 每次都是新对象、新函数
+ 组件本身非常轻
+ 比较成本大于渲染收益

## 四、为什么传对象和函数容易让 `memo` 失效？

因为 React 默认做的是浅比较。

例如:

```jsx
<Child config={{ theme: "dark" }} />
```

虽然内容一样，但每次渲染都会创建一个新的对象引用，浅比较会认为它变了。

函数同理:

```jsx
<Child onClick={() => submit(id)} />
```

每次渲染也会生成新函数。

这就是为什么 `useMemo` 和 `useCallback` 经常和 `React.memo` 搭配使用。

## 五、`useMemo` 和 `useCallback` 的优化边界

### `useMemo`

适合缓存高成本计算结果。

### `useCallback`

适合稳定函数引用，减少子组件因回调地址变化而触发的无效更新。

但要记住:

+ 不是所有函数都需要 `useCallback`
+ 不是所有计算都需要 `useMemo`
+ “优化”也有额外维护成本

## 六、列表性能优化怎么答？

这是非常常见的业务面试题。

可以从这些角度回答:

### 1. 保证 `key` 稳定

避免错误复用和额外移动。

### 2. 减少单项渲染成本

拆分子组件，避免整个列表项里做大量重复计算。

### 3. 做长列表虚拟化

比如使用:

+ `react-window`
+ `react-virtualized`

核心思想:

+ 只渲染可视区域
+ 其余项用占位高度替代

### 4. 避免父组件频繁无意义重渲染

否则再好的列表组件也可能被拖着重复执行。

## 七、如何减少 Context 带来的无效渲染？

这是中高级 React 面试常见题。

因为 `Context.Provider` 的 `value` 一变，消费它的组件通常都会重新渲染。

优化思路:

+ 拆分多个 Context，降低影响范围
+ 保持 `value` 引用稳定
+ 将频繁变动状态和静态配置分离
+ 对复杂共享状态使用更细粒度订阅的状态库

## 八、`key` 真的只影响列表吗？

大多数场景是列表，但本质不止于此。

`key` 决定的是:

+ 节点身份
+ 组件实例是否复用
+ 状态是否保留

这意味着:

+ 改变 `key` 可以强制组件重新挂载
+ 某些重置表单、重置内部状态的场景会用到它

## 九、React 项目里常见的性能问题有哪些？

常见问题包括:

+ 大组件职责过多，任何小改动都会拖着整棵子树更新
+ 在 render 中做重计算
+ 频繁创建新对象和新函数
+ Context 设计过大
+ 长列表没有虚拟化
+ 表单输入导致整页重渲染
+ 不必要的重复请求和重复状态同步

## 十、React 18 对性能优化带来了什么变化？

可以从这些点回答:

+ 自动批处理减少不必要的多次渲染
+ 并发渲染让低优先级更新可中断
+ `useTransition` 可以把部分更新标记为非紧急
+ `useDeferredValue` 可以延迟昂贵派生渲染

也就是说，React 18 的优化不只是“少渲染”，还包括“更聪明地安排什么时候渲染”。

## 十一、`useTransition` 和 `useDeferredValue` 怎么区分？

### `useTransition`

你主动告诉 React: 这部分更新不紧急。

适合:

+ 搜索筛选
+ 切页切 tab
+ 大块 UI 切换

### `useDeferredValue`

把某个值的更新延后，让昂贵依赖它的 UI 不用立刻跟上。

适合:

+ 输入框联动复杂列表
+ 实时过滤大数据集

## 十二、React 项目中的工程化优化点有哪些？

不要只谈运行时优化，工程化也常被问。

### 1. 代码分割

通过动态导入和路由级拆包降低首屏体积。

### 2. 资源按需加载

+ 图片压缩
+ 懒加载
+ 字体优化

### 3. Tree Shaking

减少无用代码进入产物。

### 4. SSR / SSG / RSC 方案选型

根据业务对首屏、SEO、交互的要求选择合适渲染策略。

### 5. 缓存策略

+ 接口缓存
+ 静态资源缓存
+ CDN

## 十三、性能优化时应该先做什么？

正确答案通常不是“先全量上 memo”。

更稳的思路是:

1. 先定位瓶颈
2. 判断是渲染问题、请求问题还是资源问题
3. 有针对性优化
4. 做前后对比验证

常见工具:

+ React DevTools Profiler
+ 浏览器 Performance 面板
+ Lighthouse

## 十四、面试高频易错点

+ 不要说虚拟 DOM 一定比原生 DOM 快
+ 不要把 `useMemo` / `useCallback` 当成默认配置
+ 不要忽略父组件重渲染对子组件的传导
+ 不要把性能优化只理解成“少写代码”

## 十五、一段适合面试现场的总结话术

> React 性能优化本质上是减少无意义渲染、降低真实 DOM 提交成本，以及在 React 18 里更合理地安排更新优先级。具体手段上，我会先分析组件为什么重渲染，再决定是否用 `React.memo`、`useMemo`、`useCallback`，同时关注 `Context` 范围、列表虚拟化、表单渲染粒度和代码分割。优化不是加几个 Hook 就结束，而是先定位瓶颈，再做有收益的调整。
