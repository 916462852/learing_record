## 组件通信、状态管理与表单

> 这部分是 React 中非常典型的业务面试题，通常会连着问: 组件怎么传值、跨层通信怎么做、Context 能不能替代 Redux、受控组件怎么理解。

<!-- more -->

## 一、React 中常见的组件通信方式有哪些？

可以从近到远来回答。

### 1. 父传子: `props`

这是最基础、最常见的方式。

特点:

+ 数据流向清晰
+ 使用简单
+ 最符合 React 单向数据流设计

### 2. 子传父: 回调函数

父组件把函数通过 `props` 传给子组件，子组件在合适时机调用这个函数，把意图或数据回传给父组件。

本质上不是“子组件直接改父组件”，而是:

> 子组件通知父组件，父组件自己决定是否更新状态。

### 3. 跨层通信: Context

适合祖先组件向深层后代共享某些公共数据，例如:

+ 主题
+ 国际化
+ 登录用户信息
+ 表单上下文

### 4. 兄弟组件通信

常见方式通常是:

+ 状态提升到共同父组件
+ 借助 Context
+ 借助全局状态管理

### 5. 全局状态管理

当多个页面、多个区域都依赖同一份状态，并且更新逻辑较复杂时，可以使用:

+ `Redux Toolkit`
+ `Zustand`
+ `Jotai`

## 二、什么是状态提升？

当两个或多个子组件需要共享同一份状态时，把这份状态提升到它们最近的共同父组件中统一管理，这就叫状态提升。

它是 React 官方非常核心的思想。

优点:

+ 数据来源唯一
+ 状态更容易同步
+ 行为更可预测

## 三、Context 是什么？适合什么场景？

一句话:

> `Context` 是 React 提供的跨层级共享数据机制，用来避免一层层手动传 `props`。

适合场景:

+ 主题
+ 语言
+ 权限信息
+ 配置项
+ 某个局部子树共享的状态

### Context 的优点

+ 解决 `props drilling`
+ 使用简单
+ 对局部共享状态很方便

### Context 的缺点

+ Provider 的 `value` 变化可能导致消费组件重新渲染
+ 如果塞入大量频繁变更的状态，维护和性能都可能变差
+ 它是“共享数据机制”，不是完整状态管理方案

所以面试里不要直接说:

> Context 可以完全替代 Redux。

更准确的回答是:

+ 小中型共享状态场景可以替代一部分全局状态管理
+ 但复杂异步流、可追踪性、中间件生态等方面，Redux 仍然有价值

## 四、Redux 的核心工作流程是什么？

经典回答:

1. 视图层派发 `action`
2. `reducer` 根据 `action` 计算新状态
3. `store` 保存最新状态
4. 订阅到状态变化的组件重新渲染

需要记住 Redux 三大原则:

+ 单一数据源
+ 状态只读
+ 使用纯函数修改状态

## 五、为什么现在更推荐 `Redux Toolkit`？

因为原始 Redux 存在这些痛点:

+ 样板代码多
+ reducer、action type、action creator 分散
+ 不小心就会写出可变更新

`Redux Toolkit` 的优势:

+ 内置 `createSlice`
+ 默认集成 Immer，写法更直观
+ 默认集成 thunk
+ 官方推荐、最佳实践统一

所以现在面试中聊 Redux，通常优先聊 `Redux Toolkit`。

## 六、什么时候该用全局状态管理，什么时候不该用？

### 适合上全局状态的情况

+ 多个页面共享
+ 多个区域反复读取
+ 更新逻辑复杂
+ 状态需要统一追踪

### 不适合上全局状态的情况

+ 只在单个组件里使用
+ 只是局部 UI 状态
+ 提升到父组件就能解决

一个常见误区是“什么都放全局”，这样会导致:

+ 状态边界模糊
+ 更新范围变大
+ 维护成本提升

## 七、服务端状态和客户端状态有什么区别？

这是近几年 React 面试里越来越常见的题。

### 客户端状态

来源于前端本地，例如:

+ 弹窗开关
+ 当前 tab
+ 输入框值
+ 本地筛选条件

### 服务端状态

来源于服务端接口，例如:

+ 用户信息
+ 列表数据
+ 详情数据
+ 分页结果

服务端状态的特点:

+ 需要请求
+ 会失效
+ 可能要缓存、重试、同步、失焦重取

所以它不太适合被当作普通全局 state 来手搓管理，这也是 `TanStack Query` 这类库流行的原因。

## 八、受控组件和非受控组件怎么回答？

### 受控组件

表单值完全由 React state 控制。

例如:

+ `value` 由 state 提供
+ `onChange` 更新 state

优点:

+ 数据流清晰
+ 便于校验、联动、回显
+ 更容易做统一管理

缺点:

+ 代码相对多
+ 复杂大表单时可能有性能压力

### 非受控组件

表单值保存在 DOM 中，通过 `ref` 获取。

优点:

+ 简单
+ 一些原生表单场景很方便

缺点:

+ 与 React 状态同步不如受控方式直接
+ 表单联动和验证通常没那么顺手

## 九、React 表单优化常问什么？

常见追问包括:

+ 大表单如何减少重渲染
+ 为什么 `React Hook Form` 性能较好
+ 表单校验该放在哪里

一个常见回答角度是:

+ 减少无关字段联动
+ 使用字段级订阅
+ 控制渲染粒度
+ 不要每次输入都让整个表单树更新

## 十、父组件如何调用子组件方法？

在函数组件体系里，常见做法是:

+ 子组件使用 `forwardRef`
+ 配合 `useImperativeHandle` 暴露有限的方法集合

这个题经常用于判断你是否理解:

+ React 主要是声明式
+ 但某些场景仍然需要命令式入口

适合的场景:

+ 表单聚焦
+ 弹窗打开关闭
+ 滚动定位

## 十一、什么时候应该用 Context，什么时候该用 Redux / Zustand？

一个很稳的回答方式:

### 更适合 Context

+ 数据层级相对固定
+ 更新频率不高
+ 只是某个局部子树共享
+ 逻辑不复杂

### 更适合 Redux Toolkit / Zustand

+ 共享范围大
+ 更新频繁
+ 逻辑复杂
+ 想把状态与业务流程更清晰地拆开

## 十二、React 组件设计有哪些建议？

这题在偏实战面试里很常见。

可以从这几个角度答:

+ 单一职责，不要一个组件包打天下
+ 业务逻辑和展示逻辑尽量分离
+ 公共逻辑提取成自定义 Hook
+ `props` 设计尽量稳定、语义清晰
+ 少暴露命令式能力，多暴露声明式接口
+ 尽量保证组件可组合

## 十三、面试高频易错点

+ 不要把 Context 说成“全局状态管理银弹”
+ 不要把所有状态都放 Redux
+ 不要混淆客户端状态和服务端状态
+ 不要把受控 / 非受控理解成“谁更高级”

## 十四、一段适合面试现场的总结话术

> React 组件通信最基础的是 `props` 和回调，跨层共享可以用 Context，更复杂的全局共享状态可以交给 `Redux Toolkit` 或其他轻量状态库。实际项目里我会先判断状态的作用域和更新复杂度，能局部解决就不全局化；同时会把服务端状态和客户端状态区分开，避免把接口数据也硬塞进传统 store。表单方面，大多数复杂联动场景还是受控组件更稳，但如果字段很多，也要关注渲染粒度和性能。
