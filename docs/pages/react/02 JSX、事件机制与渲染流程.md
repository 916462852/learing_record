## JSX、事件机制与渲染流程

> 这一篇解决 React 基础面试里的高频题: JSX 是什么、虚拟 DOM 是什么、React 怎么更新页面、事件系统怎么理解。

<!-- more -->

## 一、JSX 是什么？

一句话:

> `JSX` 是 JavaScript 的语法扩展，用来更自然地描述 UI 结构。

很多人会把 JSX 误解成模板语法，但它更准确的定位是:

+ 它最终会被编译成 JavaScript
+ 本质上是 `React.createElement` 或编译后的 `jsx` / `jsxs` 调用
+ 它只是描述 React Element 的一种更好写、更好读的方式

例如:

```jsx
const element = <div className="box">hello</div>
```

编译后大致会变成:

```js
const element = jsx("div", {
  className: "box",
  children: "hello"
})
```

## 二、React 为什么选择 JSX？

可以从三个角度回答:

### 1. 组件关注点天然聚合

在 React 里，UI 结构、状态逻辑、事件处理通常本来就是围绕组件组织的。

所以 React 团队不认为“HTML、CSS、JS 必须物理分离”才叫关注点分离，更强调:

> 组件才是更高层级的关注点边界。

### 2. 表达能力强

JSX 可以直接使用 JavaScript 表达式:

+ 条件渲染
+ 列表渲染
+ 函数调用
+ 组件组合

### 3. 工具链支持成熟

+ 代码提示更自然
+ 类型推导更友好
+ 静态分析能力更强

## 三、JSX 和模板语法的区别是什么？

可以这样理解:

+ 模板语法通常是“HTML + 指令”
+ JSX 是“JavaScript + XML 风格语法糖”

JSX 的特点:

+ 可以直接使用 JS 表达式
+ 条件、循环本质就是普通 JS
+ 学习成本更偏向 JavaScript 本身

面试里这题不要简单说“JSX 更灵活”，最好补一句:

> JSX 的核心价值是让组件中的结构和逻辑天然组合在一起，而不是额外再引入一套模板 DSL。

## 四、React Element、虚拟 DOM、真实 DOM 分别是什么？

这三个概念很容易混。

### 1. React Element

它是一个普通 JavaScript 对象，用来描述“界面上应该有什么”。

例如:

+ 节点类型是什么
+ 有哪些属性
+ 子节点是什么

### 2. 虚拟 DOM

通常可以把一棵 React Element 树理解为虚拟 DOM 的一种表现形式。

它不是浏览器里的真实节点，而是一种内存中的 UI 描述结构。

### 3. 真实 DOM

浏览器最终渲染在页面上的 DOM 节点。

更准确的关系是:

+ JSX 会生成 React Element
+ React 根据它构建和更新 Fiber 树
+ 最终在提交阶段把变化同步到真实 DOM

所以现在谈 React 更新流程时，只说“虚拟 DOM Diff”已经不够完整了，还得知道 Fiber。

## 五、React 渲染页面的大致流程是什么？

面试中可以按下面这条链路回答:

1. 编写 JSX
2. JSX 编译成 React Element
3. React 基于 Element 构建 Fiber 树
4. 调度器安排更新优先级
5. render 阶段计算需要变化的节点
6. commit 阶段把变化同步到真实 DOM

注意区分两个阶段:

### render 阶段

+ 可中断
+ 可恢复
+ 主要做计算
+ 不直接操作真实 DOM

### commit 阶段

+ 不可中断
+ 真正执行 DOM 变更
+ 执行布局相关副作用

## 六、React 事件机制是怎样的？

React 的事件不是简单地“给每个 DOM 单独绑一个原生事件”。

它有一层自己的事件系统，常见说法是“合成事件”。

### 1. 什么是合成事件？

`SyntheticEvent` 是 React 对原生事件的一层封装，目的是:

+ 抹平浏览器差异
+ 统一事件行为
+ 便于 React 控制事件触发和更新时机

### 2. 事件委托怎么理解？

历史上 React 会把很多事件统一代理到根节点，再在内部派发到对应组件。

这样做的好处:

+ 减少大量分散绑定
+ 方便框架统一管理
+ 更好地控制跨浏览器行为

在新版 React 中，事件委托的具体实现细节有演进，但“由 React 接管事件分发”这个理解仍然成立。

### 3. 为什么 React 要自建事件系统？

主要原因:

+ 统一兼容性
+ 结合批量更新机制
+ 让事件执行与框架调度更好配合

## 七、React 中事件和原生事件有什么区别？

常见区别:

+ React 事件对象是 `SyntheticEvent`
+ React 中事件名用驼峰，比如 `onClick`
+ 阻止默认行为需要显式调用 `event.preventDefault()`
+ 不能依赖返回 `false` 来阻止默认行为

面试里现在一般不必再重点强调“事件池复用”，因为新版 React 已经取消了早期那套强依赖事件池的行为细节，但你可以知道它是历史包袱之一。

## 八、列表渲染为什么一定要有 `key`？

这是 React 最高频基础题之一。

### 1. `key` 的作用

`key` 用于给同层级节点提供稳定身份标识。

React 在 Diff 时会根据 `key` 判断:

+ 哪些节点可以复用
+ 哪些节点需要新增
+ 哪些节点需要删除
+ 哪些节点需要移动

### 2. 为什么不能乱用索引当 `key`？

如果列表存在:

+ 插入
+ 删除
+ 排序

那使用索引作为 `key`` 会导致节点身份错位，进而出现:

+ 输入框值错乱
+ 局部状态串位
+ 非预期复用

### 3. 什么时候索引相对还可以接受？

只有在同时满足这些条件时才相对安全:

+ 列表是静态的
+ 不会增删改顺序
+ 不涉及局部状态错位问题

## 九、React 中条件渲染和列表渲染的本质是什么？

这题其实是在考你是否理解 JSX 最终就是 JavaScript。

### 条件渲染

本质是根据条件返回不同的 Element:

+ `if`
+ 三元表达式
+ `&&`

### 列表渲染

本质是通过 `map` 把数据映射成 Element 数组。

例如:

```jsx
list.map(item => <Item key={item.id} item={item} />)
```

## 十、受控组件和非受控组件是什么？

这个题在表单里极高频。

### 1. 受控组件

表单值由 React 状态控制。

特点:

+ 值来源于 state
+ 修改通过 `onChange`
+ React 是单一数据源

### 2. 非受控组件

表单值保存在 DOM 自身，由 `ref` 等方式读取。

特点:

+ React 不实时接管输入值
+ 适合简单表单或第三方控件封装

### 3. 怎么选？

+ 复杂联动、校验、回显场景通常优先受控组件
+ 简单输入或性能敏感场景可考虑非受控方案

## 十一、React 渲染为什么看起来比手动操作 DOM 更高效？

注意不要把这题回答成“虚拟 DOM 一定比原生 DOM 快”。

更准确的理解是:

+ 真正耗成本的是频繁、无组织地操作真实 DOM
+ React 会先在内存里计算变化，再有组织地提交更新
+ Fiber 和调度系统还能控制更新优先级，避免一次性阻塞主线程

因此优势不只是“快”，更是:

+ 可维护
+ 可预测
+ 可调度

## 十二、这一篇的面试回答模板

如果面试官问“React 渲染流程怎么理解”，可以这样答:

> React 里我们写的 JSX 会先被编译成 React Element，然后 React 会基于这些描述构建 Fiber 树。数据变化后，React 会先在 render 阶段计算出哪些节点需要更新，这个阶段是可中断的；等计算完成后，再进入 commit 阶段一次性把变更同步到真实 DOM。列表 Diff 时，`key` 用来标识节点身份，帮助 React 正确复用和移动节点。事件方面，React 也有一层自己的合成事件系统，用来统一事件行为并更好地配合更新调度。
