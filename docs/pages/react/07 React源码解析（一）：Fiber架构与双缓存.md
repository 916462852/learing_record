## React源码解析（一）：Fiber架构与双缓存

> 面试里的 React 源码题，最常见的起点就是 Fiber。你不一定要把字段名全背下来，但至少要能回答: 为什么有 Fiber、Fiber 节点长什么样、render 和 commit 为什么能拆开。

<!-- more -->

## 一、为什么 React 要引入 Fiber？

先看旧问题。

早期 React 的递归更新模型有一个明显限制:

+ 一次渲染执行到底
+ 中途不能暂停
+ 大组件树更新时容易长时间占用主线程

这会带来什么问题？

+ 输入卡顿
+ 动画掉帧
+ 高优先级交互被低优先级渲染拖慢

所以 Fiber 的本质目标不是“换个名字的虚拟 DOM”，而是:

> 把原本不可中断的渲染工作拆成可切分、可恢复、可调度的工作单元。

## 二、Fiber 到底是什么？

可以从两个角度理解。

### 1. Fiber 是一种数据结构

每个 Fiber 节点对应一个组件、原生 DOM 节点或其他 React 工作单元。

它会记录:

+ 组件类型
+ 当前 props / memoized state
+ 子节点、兄弟节点、父节点关系
+ 要执行的副作用标记
+ 更新优先级相关信息

### 2. Fiber 是一种调度执行模型

React 会按 Fiber 节点为单位逐步推进工作，而不是一口气递归到底。

这就让 React 有机会:

+ 做任务切片
+ 按优先级调度
+ 中断低优先级工作
+ 继续未完成工作

## 三、Fiber 节点的结构可以怎么理解？

即使不背源码字段，也建议知道这些核心关系:

+ `child`: 第一个子节点
+ `sibling`: 下一个兄弟节点
+ `return`: 父节点

这三个指针让 Fiber 树既能表示层级，也更适合遍历和中断恢复。

可以把它想成:

+ 不是简单的“children 数组递归”
+ 而是一种链表化的树结构

## 四、为什么 Fiber 能支持可中断渲染？

因为 React 不再要求整棵树必须一次同步完成。

它会把“处理某个 Fiber 节点”当成一个较小的工作单元，处理完后可以决定:

+ 继续做下一个
+ 暂停
+ 让出主线程
+ 之后再恢复

所以 Fiber 的价值在于:

> React 首次真正把“渲染”从一个同步递归过程，改造成了一个可被调度的过程。

## 五、什么是双缓存树？

这是源码面试里非常典型的加分点。

React 内部通常会维护两棵相关联的 Fiber 树:

+ 当前已经渲染到页面上的 `current` 树
+ 正在内存中构建的新 `workInProgress` 树

更新发生时:

+ React 基于 `current` 复制或复用节点，构建 `workInProgress`
+ 在内存里完成新一轮计算
+ commit 完成后，两者身份切换

这就是所谓的双缓存。

## 六、为什么需要 `workInProgress` 树？

因为 React 不希望一边计算、一边立刻修改真实界面。

这样设计的好处:

+ render 阶段只做计算，副作用更可控
+ 中断和重做成本更低
+ 只有确认整轮计算可提交时，才一次性进入 commit

## 七、render 阶段和 commit 阶段怎么分工？

### render 阶段

+ 构建 `workInProgress` 树
+ 计算新旧差异
+ 生成副作用信息
+ 可中断、可恢复

### commit 阶段

+ 把副作用真正提交到宿主环境
+ 更新 DOM
+ 处理 ref
+ 执行生命周期和 effect
+ 不可中断

这个拆分非常关键，因为它决定了:

+ 为什么 React 可以“算到一半先停”
+ 为什么 DOM 提交必须保持原子性

## 八、Fiber 和虚拟 DOM 的关系是什么？

面试里很多人会把这两个概念混成一个。

更准确的说法是:

+ 虚拟 DOM 更偏“UI 描述”
+ Fiber 更偏“执行和调度的数据结构”

可以理解为:

+ React Element 描述你想要什么
+ Fiber 负责让这件事可以被增量计算和调度执行

所以现代 React 面试里，如果只说“React 就是虚拟 DOM Diff”，往往不够。

## 九、Fiber 为什么是链表化树结构？

因为它更适合增量遍历和恢复工作。

如果只是普通递归树结构，中断后恢复上下文会更麻烦；Fiber 通过 `child / sibling / return` 指针，可以更灵活地推进工作流。

## 十、Fiber 节点上为什么要记录副作用？

render 阶段不会直接改 DOM，所以它需要在遍历过程中把“最终该做哪些事”记录下来。

这些事包括:

+ 插入节点
+ 更新属性
+ 删除节点
+ 处理 ref
+ 触发生命周期或 Hook effect

等到 commit 阶段，再统一执行。

## 十一、面试里怎么解释“React 更像状态机”？

因为每个 Fiber 节点都在维护某种“当前状态”和“待处理更新”，而 React 整体则在不同阶段中推进整棵树的状态转换。

例如:

+ 当前树是已提交状态
+ 新更新来到后，生成 workInProgress
+ render 计算完成后，commit 进入新状态

## 十二、Fiber 常见面试追问

### 1. Fiber 出现后是不是每次渲染都更快？

不一定。

Fiber 的核心收益不只是绝对速度，而是:

+ 可中断
+ 可调度
+ 交互更平滑

### 2. Fiber 能不能在中途随便提交 DOM？

不行。render 可以中断，但 commit 不能被随意拆碎，否则会破坏界面一致性。

### 3. Fiber 和 Hooks 有关系吗？

有。函数组件的 Hook 状态、本次更新中的 effect 信息等，最终也都挂在对应 Fiber 上参与整个渲染与提交流程。

## 十三、一段适合面试现场的总结话术

> Fiber 是 React 为了解决大组件树更新阻塞主线程而引入的一套新架构。它既是一种节点数据结构，也是可调度的执行模型。每个 Fiber 节点会记录组件类型、状态、父子兄弟关系以及副作用信息。更新时，React 会基于当前树构建一棵 `workInProgress` 树，在 render 阶段完成可中断的计算，再在 commit 阶段一次性提交 DOM 变更。双缓存树和 render/commit 分离，是理解 React 源码的核心入口。
