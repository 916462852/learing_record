## Vue3 高频面试题整理（常考）

> 这篇文章把 `Vue3` 面试里最常见、最容易被问到的题型集中整理在一起，适合查漏补缺，也适合作为面试前速记材料。为了避免误导，下面的回答尽量采用更贴近当前 Vue 3 实际行为的说法。

<!-- more -->

## 一、Vue3 相比 Vue2 有哪些核心变化？

可以从下面几个角度回答：

+ 响应式系统从 `Object.defineProperty` 升级为 `Proxy`
+ 新增 `Composition API`，逻辑复用方式更灵活
+ 支持 `<script setup>` 语法糖，单文件组件更简洁
+ 编译优化更强，比如静态提升、Patch Flag、Block Tree
+ 更好的 TypeScript 支持
+ 内置能力更现代，比如 `Fragment`、`Teleport`、`Suspense`
+ 官方推荐状态管理从 `Vuex` 逐步转向 `Pinia`

面试里一句话总结可以说：

> `Vue3` 的核心升级是“响应式重写 + 组合式 API + 编译优化 + 更好的工程化与 TS 支持”。

## 二、Composition API 和 Options API 的区别是什么？

### 1. Options API

按照 `data`、`methods`、`computed`、`watch` 等选项分散组织代码，优点是上手简单、结构直观。

### 2. Composition API

按照“功能逻辑”组织代码，把一个功能相关的状态、方法、计算属性、监听器放在一起，适合复杂组件和逻辑复用。

### 3. 面试回答重点

+ `Options API` 更适合简单组件和初学者
+ `Composition API` 更适合中大型项目、复杂逻辑拆分与复用
+ `Composition API` 对 TypeScript 更友好
+ `Vue3` 并没有废弃 `Options API`，而是提供了另一种更灵活的组织方式

## 三、为什么 Vue3 要引入 Composition API？

主要是为了解决 `Options API` 在大型组件中的几个问题：

+ 相关逻辑被拆散在 `data`、`methods`、`computed`、`watch` 中，不容易维护
+ 逻辑复用主要依赖 `mixin`，容易命名冲突、来源不清晰
+ 复杂组件在后期维护时可读性较差
+ TypeScript 类型推导不如组合式写法自然

所以 `Composition API` 的价值不只是“写法变了”，而是让代码从“按配置项分组”变成“按功能分组”。

## 四、`setup()` 和 `<script setup>` 有什么区别？

`<script setup>` 本质上是 `setup()` 的编译时语法糖。

它们的共同点：

+ 都基于组合式 API
+ 都可以使用 `ref`、`reactive`、`computed`、`watch` 等能力
+ 都没有组件实例上的 `this`

`<script setup>` 的优势：

+ 代码更少，不需要手动 `return`
+ 顶层绑定可以直接给模板使用
+ `defineProps`、`defineEmits` 等编译宏使用更自然
+ 类型推导通常更友好

面试回答可以说：

> `<script setup>` 是 Vue3 官方推荐的 SFC 写法，本质是 `setup()` 的语法糖，优势是代码更简洁、类型体验更好、模板暴露更直接。

## 五、`setup` 的执行时机是什么？为什么拿不到 `this`？

`setup` 会在组件实例创建前期执行，早于 `beforeCreate`，因此拿不到传统意义上的组件实例 `this`。

在 `setup` 中：

+ `this` 是 `undefined`
+ 可以拿到 `props`
+ 可以使用 `emit`
+ 可以注册生命周期钩子

如果面试官继续追问，可以补一句：

> `setup` 不是基于实例 `this` 工作的，而是基于闭包和组合式函数组织逻辑。

## 六、`ref` 和 `reactive` 的区别是什么？

这是 Vue3 必问题。

### 1. `ref`

+ 常用于定义单个响应式值
+ 可以包裹基本类型，也可以包裹对象
+ 在 JS 中通过 `.value` 访问
+ 在模板中会自动解包

### 2. `reactive`

+ 用于把对象、数组、`Map`、`Set` 等变成响应式代理
+ 返回的是代理对象
+ 不能直接代理基本类型
+ 在 JS 中不需要 `.value`

### 3. 怎么选？

+ 基本类型通常用 `ref`
+ 对象类型通常用 `reactive`
+ 实际项目里很多团队会“默认优先用 `ref`”，因为替换整个值时语义更统一

### 4. 面试高频补充

如果把对象传给 `ref`，对象内部同样会变成响应式对象，所以并不是“对象只能用 `reactive`”。

## 七、`ref` 为什么要用 `.value`？

因为 `ref` 本质上是一个带有 `value` 属性的包装对象，Vue 通过访问 `value` 来完成依赖收集与触发更新。

可以简单理解为：

+ `ref(1)` 返回的不是数字 `1`
+ 而是类似 `{ value: 1 }` 的响应式包装

模板中之所以不用写 `.value`，是因为 Vue 帮你做了自动解包。

## 八、`reactive` 为什么不能直接解构？

因为 `reactive` 返回的是代理对象，直接解构会把当前属性值拷贝出来，脱离原代理对象，导致响应性丢失。

错误示意：

```js
const state = reactive({ count: 0 })
const { count } = state
```

这里的 `count` 不再和原始代理保持联动。

正确思路：

+ 保持 `state.count` 这种访问方式
+ 或使用 `toRef(state, 'count')`
+ 或使用 `toRefs(state)` 后再解构

## 九、`toRef` 和 `toRefs` 的作用是什么？

+ `toRef(obj, 'key')`：把响应式对象中的某一个属性转成 `ref`
+ `toRefs(obj)`：把整个响应式对象的一层属性全部转成 `ref`

典型场景：

+ 需要解构响应式对象但又不想丢失响应性
+ 需要把某个属性单独传给组合函数

注意：

+ `toRefs` 只处理第一层属性
+ 深层对象不会继续被自动拆成多个 `ref`

## 十、Vue3 中 `props` 可以解构吗？

这个问题很容易答错。

更严谨的回答是：

+ 在早期 Vue3 版本中，直接解构 `props` 会丢失响应性
+ 这时通常需要 `toRefs(props)` 或 `toRef(props, 'xxx')`
+ 在 Vue 3.5+ 中，`<script setup>` 里支持 Reactive Props Destructure，特定写法下解构体验更好了

所以不要把这题回答成绝对的“`props` 绝对不能解构”，更准确的说法应该是：

> 要看 Vue 版本和写法；如果要兼顾稳定性与可读性，很多场景仍然会优先使用 `props.xxx` 或 `toRefs(props)`。

## 十一、`computed` 和 `watch` 的区别是什么？

### 1. `computed`

+ 用于基于已有状态派生出一个新值
+ 具有缓存能力
+ 只有依赖变化时才会重新计算
+ 更适合“一个值依赖另外一些值”

### 2. `watch`

+ 用于监听数据变化并执行副作用
+ 常用于请求接口、同步外部状态、手动操作 DOM
+ 没有返回值语义，核心是“变化后做点事”

一句话区分：

> `computed` 适合“算一个值”，`watch` 适合“做一件事”。

## 十二、`watch` 和 `watchEffect` 的区别是什么？

### `watch`

+ 需要显式指定监听源
+ 可以拿到新值、旧值
+ 可以更精确控制监听时机和范围
+ 更适合明确监听某个或某些状态

### `watchEffect`

+ 会自动收集回调中用到的响应式依赖
+ 立即执行一次
+ 更像“声明一个副作用函数”
+ 不容易直接拿到精确的旧值

面试可以这样回答：

> `watch` 是显式监听，控制更强；`watchEffect` 是自动收集依赖，写法更省，但适合更简单的副作用场景。

## 十三、`watch` 常见配置有哪些？

常考配置有：

+ `immediate: true`：立即执行一次
+ `deep: true`：深度监听
+ `flush: 'pre' | 'post' | 'sync'`：控制回调触发时机

补充一点：

+ 监听 `reactive` 对象时，很多情况下本身就会感知内部变化
+ 但当你监听的是 getter 返回值，是否需要 `deep` 要看具体写法

## 十四、Vue3 生命周期有哪些变化？

相对 Vue2，最典型的变化是：

+ `beforeDestroy` 改名为 `beforeUnmount`
+ `destroyed` 改名为 `unmounted`

组合式 API 中常用生命周期：

+ `onBeforeMount`
+ `onMounted`
+ `onBeforeUpdate`
+ `onUpdated`
+ `onBeforeUnmount`
+ `onUnmounted`
+ `onActivated`
+ `onDeactivated`

需要注意：

+ `beforeCreate` 和 `created` 没有对应的组合式钩子
+ 因为它们的大部分逻辑通常直接写在 `setup` 顶层即可

## 十五、Vue3 的响应式原理是什么？

标准回答可以分三层：

### 1. 基础能力

Vue3 使用 `Proxy` 对对象做代理，通过 `get`、`set`、`deleteProperty` 等拦截实现响应式追踪。

### 2. 核心流程

+ 读取数据时进行依赖收集
+ 修改数据时触发依赖更新
+ 最终驱动组件重新渲染或副作用重新执行

### 3. 相比 Vue2 的优势

+ 能监听属性新增和删除
+ 能更自然地监听数组索引和 `length`
+ 可以代理 `Map`、`Set`
+ 不需要像 Vue2 那样一开始就深度遍历所有属性

## 十六、为什么 `Proxy` 比 `Object.defineProperty` 更适合 Vue3？

可以从这几个点回答：

+ `Proxy` 代理的是整个对象，不是单个属性
+ 可以拦截更多操作，比如新增、删除、`in`、`ownKeys`
+ 对数组支持更自然
+ 对 `Map`、`Set` 这类结构支持更好
+ 惰性代理深层对象，初始化成本更低

当然也要知道它的缺点：

+ `Proxy` 返回的是新对象，不是原对象本身
+ 不兼容 IE

## 十七、什么是 `shallowRef`、`shallowReactive`、`markRaw`、`toRaw`？

这是进阶题，答出来会比较加分。

### `shallowRef`

只跟踪 `.value` 本身的变化，不深度追踪内部对象。

适合：

+ 大对象
+ 第三方实例对象
+ 只关心整体替换、不关心内部属性变更的场景

### `shallowReactive`

只让对象第一层变成响应式，深层对象保持原样。

### `markRaw`

标记一个对象，使其永远不被转成响应式。

常见场景：

+ 地图实例
+ 图表实例
+ 大型第三方类对象

### `toRaw`

拿到某个响应式代理对应的原始对象。

## 十八、`readonly` 和 `shallowReadonly` 有什么用？

+ `readonly`：把对象包装成只读代理，深层也只读
+ `shallowReadonly`：只限制第一层只读

常见使用场景：

+ 组合函数对外暴露状态时，只希望调用方读，不希望改
+ 避免外部误改内部状态

## 十九、`customRef` 是什么？适合什么场景？

`customRef` 允许你自己控制依赖追踪和触发更新时机。

高频场景：

+ 防抖输入框
+ 节流更新
+ 延迟触发视图更新

面试一句话回答：

> `customRef` 的价值在于把 `ref` 的追踪和触发时机交给开发者自定义控制。

## 二十、`nextTick` 是什么？什么时候用？

`nextTick` 用于等待 DOM 更新完成之后再执行回调。

因为 Vue 更新 DOM 是异步批处理的，所以修改响应式数据后，DOM 不一定会立刻同步更新。

常见场景：

+ 修改数据后获取最新 DOM
+ 列表渲染完成后执行滚动、聚焦、测量

一句话回答：

> `nextTick` 不是为了“延迟执行”，而是为了“等本轮 DOM 更新结束后再执行”。

## 二十一、`v-if` 和 `v-show` 的区别是什么？

+ `v-if` 是真正的条件渲染，会控制节点创建和销毁
+ `v-show` 只是切换元素的 `display`

如何选择：

+ 频繁切换用 `v-show`
+ 初始条件可能不成立、切换不频繁用 `v-if`

## 二十二、为什么 `v-for` 一定要配合 `key`？

`key` 的本质是给虚拟 DOM 一个稳定标识，帮助 Diff 算法正确复用或移动节点。

如果 `key` 不稳定，可能带来：

+ 列表复用错误
+ 输入框状态错乱
+ 组件局部状态串位

面试里最好补一句：

> 不推荐把数组下标当作 `key`，尤其是在增删、排序场景下。

## 二十三、Vue3 的 Diff 优化做了什么？

面试答法不要只说“更快”，而要说清楚“为什么快”。

Vue3 在编译阶段和运行时都做了优化，核心包括：

+ `Patch Flag`：标记动态内容，更新时更精准
+ `Block Tree`：把动态节点收集起来，减少无意义遍历
+ 静态提升：静态节点只创建一次
+ 事件缓存：减少不必要的事件函数重复创建

所以 Vue3 的快，不只是 Diff 算法本身快，而是“编译期已经帮运行时减少了很多工作量”。

## 二十四、什么是静态提升和 Patch Flag？

### 静态提升

不会变化的节点在编译后被提升到渲染函数外部，只创建一次，后续渲染直接复用。

### Patch Flag

编译器会给动态节点打标记，例如“这里只会变文本”“这里只会变 class”，这样更新时不必全量比较。

面试总结：

> 静态提升解决“重复创建”的问题，Patch Flag 解决“精准更新”的问题。

## 二十五、什么是 `Teleport`？

`Teleport` 可以让一段组件模板内容渲染到当前组件 DOM 层级之外的其他位置，比如 `body`。

常见场景：

+ 弹窗
+ 抽屉
+ 全局提示
+ 遮罩层

它解决的是“逻辑上属于当前组件，但视觉上需要脱离当前层级渲染”的问题。

## 二十六、什么是 `KeepAlive`？它的生命周期有哪些？

`KeepAlive` 用于缓存动态组件实例，避免组件切换时被重复卸载和重建。

常见场景：

+ 标签页切换
+ 列表页与详情页来回切换
+ 路由页面缓存

配套生命周期：

+ `onActivated`
+ `onDeactivated`

需要注意：

+ 被缓存的组件切走时不一定会触发卸载
+ 这时更适合在 `deactivated` 中做一部分清理逻辑

## 二十七、什么是 `Suspense`？

`Suspense` 用于处理异步依赖的加载态，比如异步组件或 `async setup()`。

它允许你在主内容未准备好时展示 fallback 内容。

不过在真实业务里，它的使用频率通常不如 `Teleport`、`KeepAlive` 高，面试知道它是“异步渲染协调能力”即可。

## 二十八、Vue3 组件通信方式有哪些？

常见回答：

+ 父传子：`props`
+ 子传父：`emit`
+ 兄弟/跨层：状态提升、事件转发
+ 跨层依赖注入：`provide / inject`
+ 全局状态：`Pinia`
+ 模板 `ref` + `defineExpose`：父组件主动调用子组件方法

面试里要注意：

+ 能用 `props + emit` 解决的，优先不用复杂通信
+ `provide / inject` 更适合跨层依赖，不适合滥用成全局状态

## 二十九、`provide / inject` 的使用场景是什么？

它适合祖先组件向深层后代组件传递依赖，避免一层层透传 `props`。

典型场景：

+ 表单容器与表单项
+ 主题、语言、配置注入
+ 组件库内部通信

如果面试官继续追问缺点，可以回答：

+ 数据来源不如 `props` 明确
+ 滥用后可维护性会下降
+ 更适合“依赖注入”，不适合“随处改的共享状态”

## 三十、Vue3 中父组件怎么调用子组件方法？

标准回答：

+ 父组件通过模板 `ref` 获取子组件实例
+ 子组件在 `<script setup>` 中使用 `defineExpose` 暴露方法或属性

如果子组件没用 `defineExpose`，父组件默认拿不到其内部定义的内容。

## 三十一、Vue3 中 `v-model` 有什么变化？

Vue2 默认是：

+ `value`
+ `input`

Vue3 默认变成：

+ `modelValue`
+ `update:modelValue`

同时 Vue3 支持多个 `v-model`：

```vue
<UserName
  v-model:first-name="firstName"
  v-model:last-name="lastName"
/>
```

面试里这题非常高频，最好顺带提到：

+ Vue3 中可以用 `defineModel` 简化 `v-model` 声明（新版本常见）

## 三十二、`defineProps`、`defineEmits`、`defineExpose`、`defineModel` 分别是做什么的？

它们都是 `<script setup>` 里常见的编译宏。

+ `defineProps`：声明组件 `props`
+ `defineEmits`：声明组件触发的事件
+ `defineExpose`：向父组件暴露属性和方法
+ `defineModel`：声明组件上的 `v-model`

如果还想多答一点，可以补充：

+ `defineSlots`：为插槽做类型声明
+ `defineOptions`：声明组件选项，如组件名

## 三十三、`slot` 和 `scoped slot` 怎么理解？

### 普通插槽

父组件把一段模板内容传给子组件指定位置。

### 作用域插槽

子组件在插槽渲染时，把自己的数据暴露给父组件使用。

可以理解为：

+ 普通插槽：父组件传内容
+ 作用域插槽：子组件传数据给插槽内容

## 三十四、Vue3 中如何做逻辑复用？为什么 `mixin` 不再推荐？

Vue3 更推荐用 `composable` 做逻辑复用。

也就是把一段可复用逻辑提取成 `useXxx()` 函数，比如：

+ `useMouse`
+ `useRequest`
+ `usePagination`

相比 `mixin`，`composable` 的优点是：

+ 来源更清晰
+ 命名冲突更少
+ 参数输入输出更明确
+ TypeScript 支持更好

## 三十五、Pinia 和 Vuex 的区别是什么？

高频答法如下：

+ `Pinia` 去掉了 `mutation`，修改状态更直接
+ `Pinia` 对 TypeScript 支持更友好
+ `Pinia` API 更简洁，更贴合 Vue3 的组合式思维
+ `Pinia` 支持模块化但不需要像 Vuex 那样层层嵌套
+ `Pinia` 依然支持插件、devtools、持久化等生态能力

一句话总结：

> `Pinia` 可以理解为 Vue 官方推荐的新一代状态管理方案，语法更轻、类型更强、心智负担更低。

## 三十六、Vue3 性能优化可以从哪些方面回答？

可以从“框架层”和“业务层”两部分回答。

### 1. 框架层

+ 编译优化：静态提升、Patch Flag、Block Tree
+ 更现代的响应式系统
+ 更好的 Tree Shaking

### 2. 业务层

+ 合理拆分组件，缩小更新范围
+ 列表渲染保证 `key` 稳定
+ 避免模板中写过重计算逻辑
+ 用 `computed` 缓存派生状态
+ 谨慎使用深度 `watch`
+ 大对象或第三方实例可用 `shallowRef` / `markRaw`
+ 路由级、组件级按需加载
+ 图片、资源、首屏加载策略优化

## 三十七、为什么说 Vue3 对 TypeScript 更友好？

因为 Vue3 的组合式 API 天然更接近普通函数组合，类型推导路径更直接。

典型体现：

+ `props`、`emit` 类型声明更自然
+ 组合函数返回值更容易复用类型
+ `Pinia`、组件、hooks 的类型推导体验更好
+ `<script setup lang="ts">` 的开发体验更顺畅

## 三十八、面试时 Vue3 常见易错点有哪些？

这个部分很适合拿来收尾。

+ 不要把 `watch` 和 `computed` 混着讲
+ 不要把 `props` 解构问题答成绝对结论，要看版本和写法
+ 不要说 Vue3 “完全没有双向绑定”，`v-model` 只是语法糖，但场景确实存在
+ 不要说 `setup` 里还能像 Vue2 一样用 `this`
+ 不要把 `provide/inject` 当成全局状态管理
+ 不要只背“Vue3 更快”，要能说出编译优化和响应式升级的原因

## 三十九、一段适合面试现场的总结话术

如果面试官问：“你怎么理解 Vue3？”

你可以这样总结：

> `Vue3` 相比 `Vue2` 最大的升级，一是用 `Proxy` 重写了响应式系统，二是引入了 `Composition API` 来提升大型组件的逻辑组织和复用能力，三是在编译阶段做了很多优化，比如静态提升、Patch Flag、Block Tree，所以运行时更新更精准。同时，`Vue3` 对 TypeScript、`<script setup>`、状态管理和工程化体验也更友好，所以它不只是 API 更新，而是整套开发体验都更现代了。
