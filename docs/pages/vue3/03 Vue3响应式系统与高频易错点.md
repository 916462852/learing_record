## Vue3响应式系统与高频易错点

> 这部分是 Vue3 面试里最核心的一条主线。很多题表面上在问 `ref`、`reactive`，本质上都在考你对 Vue3 响应式模型的理解。

<!-- more -->

## 一、Vue3 的响应式系统是什么？

一句话:

> Vue3 通过 `Proxy`、依赖收集和触发更新机制，把状态变化自动映射到视图和副作用执行上。

它的核心流程可以简单理解为:

+ 读取数据时收集依赖
+ 修改数据时触发依赖
+ 相关组件或副作用重新执行

## 二、`ref` 和 `reactive` 的区别是什么？

### `ref`

+ 更适合单个响应式值
+ 可以包裹基本类型，也可以包裹对象
+ 在 JS 中通过 `.value` 访问
+ 在模板中通常会自动解包

### `reactive`

+ 更适合对象、数组、`Map`、`Set`
+ 返回代理对象
+ JS 中不需要 `.value`
+ 不能直接代理基本类型

## 三、`ref` 为什么要用 `.value`？

因为 `ref` 返回的不是原始值本身，而是一个带 `value` 属性的响应式包装对象。

Vue 借助这个包装对象:

+ 收集依赖
+ 触发更新

所以面试里可以说:

> `.value` 不是语法负担本身，而是 `ref` 这一层响应式包装的访问入口。

## 四、对象到底该用 `ref` 还是 `reactive`？

更稳的回答不是非黑即白，而是:

+ 单值状态常用 `ref`
+ 结构化对象常用 `reactive`
+ 但对象也可以被 `ref` 包裹

很多团队在实际项目里会偏向“默认优先用 `ref`”，因为整体替换时语义更统一；也有团队对象默认更爱用 `reactive`。这本身不是唯一答案。

## 五、为什么 `reactive` 直接解构容易丢失响应性？

因为 `reactive` 返回的是代理对象，直接解构会把当前值拷贝出来，后续这个局部变量就不再和原代理对象保持联动。

例如:

```js
const state = reactive({ count: 0 })
const { count } = state
```

这里的 `count` 已经不再是响应式引用。

## 六、`toRef` 和 `toRefs` 有什么用？

这两个 API 的核心作用就是:

> 在“需要解构”的前提下，尽量保住响应性。

### `toRef`

把对象上的单个属性转成 `ref`。

### `toRefs`

把对象第一层属性全部转成 `ref`。

高频提醒:

+ `toRefs` 只处理第一层
+ 深层对象不会被自动继续拆开

## 七、为什么 Vue3 用 `Proxy` 替代 `Object.defineProperty`？

这是 Vue3 高频原理题。

主要原因:

+ `Proxy` 代理整个对象，不是单个属性
+ 能拦截新增、删除、`in`、`ownKeys` 等更多操作
+ 对数组、`Map`、`Set` 更自然
+ 深层对象可以惰性代理，不需要一上来深度遍历全部属性

## 八、`Proxy` 相比 `defineProperty` 的优势怎么答？

可以从这四个方向讲:

### 1. 监听范围更完整

新增、删除、数组索引、`length` 变化都更好处理。

### 2. 适配数据结构更多

不仅对象和数组，还包括 `Map`、`Set`。

### 3. 初始化成本更低

不需要像 Vue2 那样一开始递归劫持所有属性。

### 4. 实现模型更现代

配合 `Reflect` 更符合底层对象操作语义。

## 九、Vue3 响应式里的依赖收集怎么理解？

读取数据时，Vue 会记录“谁正在使用这个值”。

修改数据时，Vue 会找到依赖它的副作用函数或组件更新逻辑，再重新触发。

更底层一点看，可以理解成:

+ `track` 负责收集依赖
+ `trigger` 负责触发依赖

## 十、`computed` 为什么有缓存？

因为 `computed` 本质上是基于响应式依赖的惰性求值。

只要依赖没变:

+ 不会重新计算
+ 会复用上次结果

这也是它和普通方法调用的重要区别。

## 十一、`watch` 和 `watchEffect` 在响应式层面有什么区别？

### `watch`

+ 显式声明监听源
+ 控制更精确
+ 可以拿到新值和旧值

### `watchEffect`

+ 自动收集回调中使用到的响应式依赖
+ 首次会立即执行
+ 更像一个响应式副作用函数

## 十二、什么是 `shallowRef`、`shallowReactive`、`markRaw`、`toRaw`？

这类题属于进阶加分项。

### `shallowRef`

只跟踪 `.value` 本身变化，不深度追踪内部对象。

### `shallowReactive`

只让对象第一层变成响应式。

### `markRaw`

标记对象，永远不要把它变成响应式。

### `toRaw`

拿到代理对象背后的原始对象。

典型场景:

+ 图表实例
+ 地图实例
+ 大型第三方对象

## 十三、`readonly` 和 `shallowReadonly` 的作用是什么？

它们的价值在于“限制修改边界”。

+ `readonly`：深层只读
+ `shallowReadonly`：只限制第一层只读

常见场景:

+ composable 对外暴露只读状态
+ 防止外部误改内部响应式数据

## 十四、Vue3 响应式里最常见的易错点

+ 直接解构 `reactive`
+ 把 `props` 解构问题说成绝对结论
+ 忽视 `ref` 在 JS 里需要 `.value`
+ 监听 getter 时误判是否需要 `deep`
+ 误以为 `watchEffect` 能完全替代 `watch`

## 十五、一段适合面试现场的总结话术

> Vue3 的响应式系统基于 `Proxy` 和依赖收集机制。读取状态时会收集依赖，修改状态时会触发相关副作用和组件更新。日常开发中最常用的是 `ref` 和 `reactive`：`ref` 更适合单值，`reactive` 更适合对象代理，但两者不是绝对对立的。很多高频面试题，比如 `toRef`、`toRefs`、`watchEffect`、`shallowRef`、`markRaw`，本质上都可以回到“依赖怎么收集、状态怎么触发、响应性怎么保持”这条主线上理解。
